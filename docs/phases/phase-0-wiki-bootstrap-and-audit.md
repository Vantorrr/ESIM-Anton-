# Phase 0: Wiki Bootstrap & Legacy Audit

> [Корневой документ wiki](../README.md)

## Цель

Собрать достоверную wiki по унаследованному проекту и отделить подтвержденную архитектуру от устаревших описаний.

## Результат

- `docs/architecture/` переведена из шаблона в рабочую project wiki.
- Зафиксирован аудит корневых markdown-файлов против текущего кода.
- Обновлён entrypoint `docs/README.md` и roadmap в `docs/phases/README.md`.

## Оценка

Небольшая аналитическая фаза без кодовых изменений runtime, но с высокой ценностью для следующих сессий.

## Зависит от

- доступа к репозиторию;
- чтения `backend`, `admin`, `client`, `bot`, `shared`;
- чтения корневых markdown-документов.

## Пререквизиты

- сначала прочитать `docs/README.md`;
- затем проверить код, а не доверять старой документации.

## Архитектурные решения

- Source of truth смещён в сторону кода и новой wiki.
- Корневые документы считаются legacy-слоем, а не актуальной инженерной спецификацией.
- Для нового контекста проект описывается как monorepo с модульным backend API и несколькими клиентами.

## Шаги (журналы)

### Шаг 1. Инвентаризация входных документов

### Цель

Собрать все существующие markdown-источники и понять, какие из них относятся к архитектуре, запуску, интеграциям и отчётам.

### Что нужно сделать

- проверить `docs/README.md`, `docs/architecture/README.md`, `docs/phases/README.md`;
- прочитать тогда ещё корневые `README.md`, `ARCHITECTURE.md`, `SETUP.md`, `DEPLOYMENT.md`, `PROJECT_SUMMARY.md`, интеграционные markdown-файлы.

### Результат шага

Выявлено, что корневые документы описывают разные стадии проекта и конфликтуют между собой.

### Шаг 2. Сверка документации с кодом

### Цель

Проверить, что реально реализовано в кодовой базе.

### Что нужно сделать

- проверить структуру пакетов `backend/admin/client/bot/shared`;
- подтвердить backend modules, Prisma schema, controllers, auth/payment/provider flows;
- подтвердить реальные страницы `client` и `admin`.

### Результат шага

Подтверждено, что проект существенно шире старых описаний: есть отдельный `client`, multi-auth, CloudPayments, top-up flow, cron monitoring и promo codes.

### Шаг 3. Сборка новой architecture wiki

### Цель

Сделать компактный набор markdown-файлов, с которого новая сессия сможет быстро войти в проект.

### Что нужно сделать

- обновить `docs/architecture/README.md`;
- создать verified overview, module map, runtime notes и legacy audit;
- зафиксировать gotchas и локальные правила поддержки.

### Результат шага

`docs/architecture/` стала рабочей wiki вместо шаблона.

## Верификация

- документы в `docs/architecture/` существуют и связаны индексом;
- `docs/README.md` ведёт в актуальные документы, а не в шаблон;
- `docs/phases/README.md` отражает уже выполненную audit phase;
- ключевые утверждения в wiki подтверждаются кодом из `backend`, `admin`, `client`, `bot`.
