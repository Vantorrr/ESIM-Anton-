## Contract Notes

- `productsApi.getAll()` был единственной подтверждённой legacy ambiguity в админке: экран читал `response.data`, `response.data.data` и `response.data.products`. Backend возвращает прямой массив из `ProductsService.findAll()`, поэтому Step 01 нормализует frontend на прямой `AdminProduct[]` и убирает fallback-ветки.
- `dashboardApi.getStats()` и `analyticsApi.getDashboard()` бьют в один и тот же endpoint. В рамках совместимости обе обёртки сохранены, но обе типизированы одним и тем же `DashboardStats`.
- `systemSettingsApi` intentionally mixed: read endpoints отдают direct object/map, write endpoints возвращают `{ success, data }` или `{ success, message }`. Это подтверждённый backend contract, не frontend inconsistency.
- `ordersApi.getAll()` возвращает reconciliation snapshot как часть admin list contract. Это backend-derived field из `OrdersService.decorateOrderWithReconciliation()`, не локальный frontend computed state.
- Для `dedupe()` backend `report` не имеет стабильного отдельного DTO в shared-слое и пока не потребляется экраном. В Step 01 он сохранён как `unknown[]` с явной compatibility note вместо догадки о внутренней структуре.
